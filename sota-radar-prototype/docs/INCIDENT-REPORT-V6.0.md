## 📋 事故摘要

V6.0 版本部署至 MiniMax Space 后，用户访问页面得到**深色空白页**——页面完全无法渲染。经确认为 `api.ts` 中硬编码的 `API_BASE = 'http://localhost:7861'` 导致：浏览器无法访问服务器内部地址，所有 API 调用均告失败，且 fallback 逻辑未能有效保护，页面核心数据加载受阻。

## 🔍 根因分析

### 直接原因

`api.ts` 第 3 行：
```typescript
const API_BASE = 'http://localhost:7861'
```
`http://localhost:7861` 是**服务器内部地址**，浏览器在 MiniMax Space 公网环境下无法访问。所有依赖 `fetchModels()` 的组件初始化时网络请求全部失败。

### 根本原因（架构层）

前端与后端的**架构边界混乱**：

```
设计预期：
  前端（静态 SPA） ──→ API ──→ 后端（服务器内部）
                              ↑ 正确：同服务器内部调用

实际情况：
  前端（公网 SPA） ──✗──→ localhost:7861（不可达）
                              ↑ 错误：公网浏览器无法访问内部地址
```

V6.0 将"服务端内部调用"错误地暴露给了"公网浏览器"，违反了基本的前后端分离原则。

### 直接触发因素

| 问题点 | 说明 |
|--------|------|
| 硬编码端口 | `localhost:7861` 被直接写入前端 Bundle，无法随环境变化 |
| Fallback 未生效 | `fetchModels()` 的 try/catch fallback 未被正确执行（模型过滤逻辑缺失） |
| 质检盲区 | 质检流程未检查"API 地址是否公网可达"，仅验证代码语法 |
| 发布前未验证 | V6.0 发布前未在真实部署环境（MiniMax Space）进行端到端测试 |

## 🔧 修复方案

### 立即修复（01:55，~5 分钟）

- [x] `API_BASE` 回滚至 `http://localhost:7860`（仍为内部地址，但绕过 7861 端口的未初始化问题）
- [x] 生产环境强制使用静态数据（`IS_PRODUCTION` 检测 + 直接 `filterStaticModels`）
- [x] `triggerFlywheel` 和 `fetchStatus` 在生产环境返回 mock 数据，不发送网络请求
- [x] `fetchModels` 生产路径：直接调用 `filterStaticModels`，绕过所有网络调用

### 长期整改（待排期）

- [ ] **后端公网暴露**：将后端部署到公网可访问服务器（如 `api.sotaradar.ai`），前端使用 HTTPS API
- [ ] **API Health Check**：前端启动时检测 API 可达性，若不可达则明确提示"数据来源于静态备份"
- [ ] **发布 Checklist**：每次前端部署前必须通过真实环境 E2E 验证（Playwright 截图检查）
- [ ] **CI/CD 门禁**：CI 中检测 `api.ts` 等文件是否包含 `localhost`，发现则阻断发布
- [ ] **明确架构文档**：`api.ts` 增加架构注释，说明生产环境不调用 API 的设计意图
