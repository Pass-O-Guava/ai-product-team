# multi-modal-llm-kb-setup

> Skill 版本：v1.0 | 创建日期：2026-04-04 | 适用场景：多模态大模型知识库从零搭建

---

## 1. 名称与描述

**名称：** `multi-modal-llm-kb-setup`

**一句话描述：** 标准化地从零建立多模态大模型知识库，包括目录结构、核心文档、评估体系和数据索引。

**适用场景：**
- 新建一个面向开发者/企业的 AI 模型知识库
- 为 SOTA 模型收录项目搭建规范化框架
- 建立跨模态（文字/视觉/音频/视频/多模态统一）模型索引体系
- 沉淀多模态 AI 领域的结构化对比数据

## 2. 适用条件

当遇到以下任一需求时，激活本 Skill：

- 用户说：「帮我建一个 SOTA 模型知识库」

## 3. 标准化流程（5步法）

### Step 1：明确产品定位（README.md 核心）

在动手建文件之前，先明确：

```
- 这个知识库是什么？（产品定义）
- 解决什么问题？（用户价值）
- 数据来源是什么？（来源可信度）
- 谁是目标用户？（决定内容深度）
```

输出 → `/项目名/README.md`（产品定位 + 使用指南 + 更新机制 + 贡献指南）

---

### Step 2：设计评估维度体系

建立一套客观、可量化的对比标准：

```
必须维度：
✅ 能力表现（基准测试分数）
✅ 推理速度（延迟/吞吐量）
✅ 许可证（商用合规）
✅ 算力需求（显存/内存）
✅ 易用性（社区生态/文档）
✅ 多模态支持（多模态场景）

SOTA 判断规则：
- 维度性（非全局性）：一个模型不可能所有维度都是 SOTA
- 基准测试最高分 + 显著性阈值（>3%）
- 发布时效性：新模型刷新自动降级旧 SOTA
```

输出 → `/项目名/compare/methodology.md`

---

### Step 3：设计分类索引结构

按模态划分一级目录，每个模态下按任务类型二级索引：

```
models/
├── README.md              ← 总索引表（表格形式）
├── text/                  ← 文字模型（LLM / Code / Math）
├── vision/                ← 视觉模型（分类/检测/生成）
├── audio/                 ← 音频模型（ASR/TTS/情感）
├── video/                 ← 视频模型（理解/生成）
└── multimodal/           ← 多模态统一模型
```

索引表格式：Markdown 表格（模型名 | 核心指标 | 许可证 | SOTA维度 | 数据来源）

输出 → `/项目名/models/README.md` + 各子目录占位符

---

### Step 4：建立更新机制

知识库的核心价值在于「时效性」，必须设计好更新机制：

```
- 定期扫描：HuggingFace Trending / ModelScope 新模型
- 版本追踪：已收录模型的版本迭代
- 社区贡献：PR 通道 + 贡献指南
- 自动化提醒：基于模型发布动态触发更新
- SOTA 刷新规则：触发条件 + 标记格式
```

## 5. 本次沉淀的要点

### ✅ 做得好的

1. **定位清晰再动手**：先写 README 明确产品定位，避免后续结构混乱
2. **六维度评估体系**：从能力/速度/许可证/算力/易用性/多模态六个维度全面评估，覆盖真实选型决策需求
3. **SOTA 维度性而非全局性**：强调 SOTA 是按维度判断的，避免「全能冠军」误区
4. **数据来源透明度**：明确标注来源优先级和可靠性评级，确保数据可信度
5. **索引表设计**：总索引（models/README.md）+ 快速检索入口，用户体验优先

### ⚠️ 需要注意的坑

1. **不要在初始版本收录假数据**：v0.1 阶段各模型卡片标注「（建设中）」，等真实数据到位再填入
2. **许可证信息必须可验证**：仅收录 HuggingFace / ModelScope 官方标注的许可证，不凭猜测
3. **评估维度须与用户决策相关**：不是越多越好，聚焦用户真正关心的维度（商用合规 > 极端能力）
4. **更新机制要落到执行层**：不能只写「定期更新」，要说明更新频率和数据来源

### 🔄 可复用的模式
